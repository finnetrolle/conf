# План разработки браузерного приложения для 1:1 видеосессий

## 1. Цель проекта

Собрать максимально простой `MVP` приложения для видеосвязи между двумя пользователями, которое:

- работает полностью в браузере, без `Flutter` и без нативных клиентов;
- позволяет создать одноразовую сессию и передать ссылку второму участнику;
- дает обоим участникам аудио- и видеосвязь через браузер;
- завершает сессию автоматически, когда оба участника вышли;
- остается достаточно простым, чтобы быстро дойти до работающего результата.

Для первой версии мы сознательно убираем все, что не нужно для сценария "создал ссылку -> открыл ссылку -> созвонились".

## 2. Главный пользовательский сценарий

Ниже фиксируется целевой клиентский путь для `MVP`.

1. Пользователь 1 заходит на сайт приложения и нажимает кнопку `Создать сессию`.
2. У пользователя 1 открывается окно видеосвязи, где он может настроить камеру, микрофон и источник звука. В этом окне уже есть локальное видео, и пользователь 1 видит себя как подключившегося участника.
3. В этом же окне пользователь 1 видит кнопку `Ссылка на сессию`. При нажатии копируется ссылка для подключения ко встрече.
4. Пользователь 1 передает ссылку пользователю 2 через любой внешний канал, например `SMS`, `Telegram` или email.
5. Пользователь 2 открывает ссылку и сразу попадает в ту же сессию. Он может переключить микрофон и камеру и видит встречу.
6. Пользователи 1 и 2 общаются через браузер по видеосвязи. Когда оба участника покидают сессию, она завершается.

## 3. Что меняется относительно исходного плана

Из исходного плана убираем:

- `Flutter`;
- нативные клиенты под `Windows`, `macOS`, `Android`;
- `NestJS`;
- обязательную `PostgreSQL` для `MVP`;
- регистрацию, логин, список пользователей и адресную книгу;
- входящие звонки по user id;
- сущности `users`, `devices`, `calls`, `call_events` как обязательную часть первой версии;
- все фичи, не влияющие напрямую на сценарий "создать ссылку и созвониться".

Оставляем только то, что реально нужно:

- браузерный клиент;
- signaling-сервер;
- `WebRTC`;
- `STUN/TURN` через `coturn`;
- минимальное управление жизненным циклом сессии;
- базовые логи и диагностику.

## 4. Выбранная стратегия

Для первой версии берем следующий стек:

- клиент: `TypeScript` + `React` + `Vite` + `shadcn/ui`;
- видеосвязь: браузерный `WebRTC`;
- signaling и backend: `Kotlin` + `Ktor` + `WebSocket`;
- NAT traversal: `coturn`;
- хранение состояния `MVP`: in-memory на сервере;
- локальный запуск: `docker-compose`.

Почему так:

- браузерный клиент сразу покрывает сценарий "открыл сайт и подключился", без установки приложения;
- `React` + `shadcn/ui` дают быстрый и аккуратный UI без тяжелого нативного слоя;
- `Kotlin` на сервере полностью подходит для этой задачи, а `Ktor` хорошо закрывает `HTTP`, `WebSocket` и простой signaling;
- для `1:1` звонков подходит `P2P WebRTC`, без `SFU`;
- для первой версии база данных не обязательна, потому что сессии одноразовые и могут жить только в памяти сервера;
- `coturn` остается обязательным, иначе звонки будут нестабильны в реальных сетях;
- запуск через `docker-compose` упрощает локальную сборку и подъем всех сервисов одной командой.

## 5. Архитектурное решение

### 5.1 Общая схема

```text
Browser Client A  <-- media via WebRTC -->  Browser Client B
       |                                         |
       +------ signaling via WebSocket ----------+
                         |
                  Kotlin Ktor Server
                         |
                 In-memory session state

Both clients <------ ICE/STUN/TURN ------> coturn
```

### 5.2 Роли компонентов

#### Браузерный клиент

Отвечает за:

- создание сессии;
- подключение к сессии по ссылке;
- запрос доступа к камере и микрофону;
- локальное превью;
- показ удаленного видео;
- выбор камеры и микрофона;
- выбор аудиовыхода там, где браузер это поддерживает;
- `mute/unmute`;
- `camera on/off`;
- обмен signaling-сообщениями;
- корректный выход из сессии и освобождение ресурсов.

#### Сервер на Kotlin

Отвечает за:

- создание сессии;
- генерацию безопасной ссылки подключения;
- проверку, существует ли сессия и есть ли в ней место для второго участника;
- signaling через `WebSocket`;
- выдачу ICE-конфигурации;
- отслеживание подключений и отключений участников;
- завершение и очистку сессии, когда участников не осталось;
- минимальные логи и healthcheck.

#### TURN/STUN

Отвечает за:

- прохождение `WebRTC` в NAT-сценариях;
- fallback на relay, если прямое соединение невозможно.

## 6. Минимизация сложности

Чтобы не перегружать первую версию, фиксируем следующие ограничения:

- только `1:1`, максимум два участника в сессии;
- без аккаунтов и логина;
- без постоянного хранения истории;
- без нативных приложений;
- без входящих вызовов и статуса присутствия пользователей;
- без `PostgreSQL`, если не появится жесткая потребность по ходу реализации;
- без `Redis` на старте, если сервер запускается в одном экземпляре.

Если позже потребуется масштабирование на несколько инстансов, тогда можно добавить:

- `Redis` для shared session state и pub/sub;
- постоянное хранилище для истории и аналитики;
- отдельный reverse proxy и полноценное production-окружение.

## 9. Ограничения браузера, которые надо зафиксировать сразу

Есть несколько технических ограничений, которые нельзя игнорировать в плане:

- подключение "сразу" после открытия ссылки означает, что страница автоматически начинает `getUserMedia` и signaling; если разрешения на камеру или микрофон еще не выданы, браузер покажет системный prompt;
- выбор аудиовыхода (`speaker selection`) работает не во всех браузерах одинаково;
- на практике для `MVP` лучше считать основными браузерами последние версии `Chrome` и `Edge`;
- поддержка `Safari` и `Firefox` возможна, но ее лучше считать отдельной задачей стабилизации, а не скрытым обязательством первой версии.

Следствие для плана:

- основной целевой профиль `MVP`: `Chrome/Edge` на `Windows`, `macOS`, `Android`;
- поддержку других браузеров уточняем после рабочего `MVP`.

## 10. Модель сессии

Для первой версии вместо модели пользователей и звонков берем модель одноразовой сессии.

### Сущность `session`

Минимальные поля:

- `session_id`
- `share_token`
- `created_at`
- `status`
- `participants`
- `last_activity_at`

### Сущность `participant`

Минимальные поля:

- `participant_id`
- `session_id`
- `connection_id`
- `joined_at`
- `left_at` nullable
- `media_state`

### Статусы сессии

- `created` - сессия создана, в ней только пользователь 1;
- `waiting_for_peer` - первый участник уже внутри и ждет второго;
- `connecting` - второй участник зашел, идет `WebRTC` negotiation;
- `active` - оба участника подключены;
- `ended` - оба участника вышли;
- `expired` - сессия удалена по TTL или cleanup policy.

### Правила жизненного цикла

- ссылка должна содержать криптографически стойкий идентификатор или секрет, чтобы ее нельзя было угадать;
- в сессию можно подключить не больше двух участников;
- если один участник вышел, второй остается в сессии;
- когда последний участник выходит или закрывает вкладку, сервер завершает сессию;
- завершенная сессия больше не должна принимать новые подключения;
- для защиты от мусора нужна фоновая очистка старых и зависших сессий.

## 11. API и signaling

Для `MVP` разделяем обычные `HTTP` endpoints и signaling через `WebSocket`.

### HTTP

Минимально нужны:

- `POST /api/sessions` - создать сессию;
- `GET /api/sessions/{sessionId}` - получить метаданные и проверить, можно ли подключиться;
- `GET /api/ice-servers` - получить `STUN/TURN` конфигурацию;
- `GET /health` - healthcheck.

Ожидаемое поведение:

- `POST /api/sessions` создает сессию, возвращает `session_id`, `share_url`, `participant_token` для создателя;
- `GET /api/sessions/{sessionId}` нужен для страницы по ссылке, чтобы быстро понять статус сессии до старта signaling;
- `share_url` должен быть достаточным для входа второго участника без логина.

### WebSocket события

Минимальные типы событий:

- `session.join`
- `session.ready`
- `session.full`
- `participant.joined`
- `participant.left`
- `webrtc.offer`
- `webrtc.answer`
- `webrtc.ice_candidate`
- `webrtc.restart_ice`
- `media.state_changed`
- `error`

### Направление событий

1. Пользователь 1 нажимает `Создать сессию`.
2. Клиент вызывает `POST /api/sessions`.
3. Сервер возвращает `share_url` и данные для подключения создателя.
4. Клиент открывает экран сессии, показывает локальное превью и подключается к `WebSocket`.
5. Пользователь 2 открывает `share_url`.
6. Страница пользователя 2 получает данные сессии, запрашивает медиа и подключается к `WebSocket`.
7. Сервер сообщает первому участнику, что второй вошел: `participant.joined`.
8. Участники обмениваются `offer/answer/ice_candidate`.
9. После установления связи начинается обмен медиа напрямую или через `TURN`.
10. При выходе участника сервер шлет `participant.left`, а после выхода последнего участника закрывает сессию.

## 12. Экран и UX сессии

Экран видеосвязи должен покрывать весь основной сценарий без переходов по дополнительным страницам.

### Обязательные элементы экрана

- блок локального видео;
- блок удаленного видео или заглушка `Ожидание второго участника`;
- селектор камеры;
- селектор микрофона;
- селектор аудиовыхода там, где поддерживается;
- кнопка `Вкл/выкл микрофон`;
- кнопка `Вкл/выкл камеру`;
- кнопка `Ссылка на сессию`;
- кнопка `Покинуть сессию`;
- индикатор состояния подключения.

### Обязательное поведение

- пользователь 1 после создания сессии сразу оказывается внутри нее;
- пользователь 1 видит свое видео до подключения второго участника;
- копирование ссылки должно работать одной кнопкой;
- пользователь 2 по ссылке попадает сразу на экран сессии, а не на отдельный flow логина;
- при отказе в доступе к камере/микрофону UI должен объяснить, что нужно сделать;
- при разрыве соединения должно быть понятное состояние, а не "тихий" зависший экран.

Цель: довести соединение до реального аудио и видео между двумя браузерами.

- создать `RTCPeerConnection`;

Критерий готовности:

- два браузера видят и слышат друг друга в локальной сети;
- повторное подключение к новой сессии возможно без перезапуска приложения;
- выход из сессии освобождает медиа-ресурсы.

## Этап 5. TURN/STUN и реальные сети

Цель: сделать звонки рабочими за пределами одной локальной сети.

Задачи:

- поднять `coturn`;
- настроить `STUN/TURN`;
- подключить выдачу `ICE` конфигурации из сервера;
- протестировать сценарии `Wi-Fi -> LTE`, домашний NAT и офисная сеть;
- логировать, когда соединение идет напрямую, а когда через relay.

Критерий готовности:

- соединение стабильно поднимается в разных сетях;
- при невозможности прямого канала используется `TURN`;
- конфигурация `ICE` не зашита жестко в клиенте.

## 15. Ключевые технические решения

### Решение 1. Полностью браузерный клиент

Причина:

- это напрямую соответствует желаемому пользовательскому сценарию;
- не нужен `Flutter` и нативные сборки;
- меньше стоимость разработки и поддержки;
- ссылку можно открыть мгновенно на любой поддерживаемой платформе.

### Решение 2. Сервер на Kotlin

Причина:

- `Kotlin` здесь полностью уместен;
- `Ktor` достаточно легкий для `HTTP` + `WebSocket` + signaling;
- серверная часть остается типобезопасной и компактной.

### Решение 3. Без базы данных в первом MVP

Причина:

- первая версия не хранит пользователей, историю и постоянные сущности;
- сессии одноразовые и завершаются автоматически;
- in-memory хранение заметно упрощает реализацию.

Ограничение:

- на старте предполагается один экземпляр сервера;
- после появления требований к масштабированию нужно будет добавить shared state.

### Решение 4. `P2P WebRTC` для первой версии

Причина:

- всего два участника;
- сервер не занимается медиамаршрутизацией;
- меньше инфраструктурной и операционной сложности.

### Решение 5. `TURN` обязателен с самого начала

Причина:

- без `TURN` реальный интернет-сценарий будет ненадежным;
- проблемы всплывут слишком поздно, если оставить это "на потом".

## 16. Риски

### Технические риски

- неполная кроссбраузерность `WebRTC` и device APIs;
- ограниченная поддержка выбора аудиовыхода;
- race conditions в signaling flow;
- неочевидные сценарии отключения вкладки и повторного входа;
- нестабильная работа без корректного `TURN`.

### Продуктовые риски

- попытка вернуть в `MVP` логин, профили и историю слишком рано;
- скрытое ожидание поддержки всех браузеров без отдельного этапа стабилизации;
- переусложнение сервера хранилищами и очередями до появления реальной необходимости.

## 17. Definition of Done для MVP

`MVP` считается завершенным, когда:

- пользователь 1 открывает сайт и нажимает `Создать сессию`;
- пользователь 1 сразу попадает в экран видеосвязи и видит себя;
- ссылка на сессию копируется одной кнопкой;
- пользователь 2 открывает ссылку и попадает в ту же сессию;
- оба пользователя видят и слышат друг друга в браузере;
- оба могут управлять микрофоном и камерой;
- сервер корректно завершает сессию после выхода обоих участников;
- приложение работает в целевых браузерах через интернет, а не только в локальной сети;
- проект поднимается по документированным шагам из одного репозитория.
